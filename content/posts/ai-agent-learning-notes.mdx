## 一、Workflow vs Agent：电商退款场景的两种方案

### 1.1 场景设定

某电商公司处理售后退款申请，有两种技术路线可选：

```
方案 A (Workflow)：固定规则流程
方案 B (Agent)：   AI 智能体自主决策
```

### 1.2 两种方案的决策逻辑对比

**方案 A —— 固定 Workflow：**

```
用户提交退款
    │
    ├── 特殊商品（定制品）？──→ 一律拒绝
    │
    └── 一般商品
         │
         ├── 7天内
         │    ├── <100元  ──→ 自动通过
         │    ├── 100-500元 ──→ 客服审核
         │    └── >500元  ──→ 主管审批
         │
         └── 超过7天
              └── 无论金额 ──→ 客服审核 / 主管审批
```

**方案 B —— 智能 Agent：**

```
用户提交退款
    │
    └── Agent 综合评估
         ├── 理解退款政策
         ├── 分析用户历史行为（信用、退货频率）
         ├── 评估商品状况（破损证据、物流记录）
         └── 自主决策 → 通过 / 拒绝 / 转人工
```

### 1.3 优缺点对比

| 维度 | 方案 A：固定 Workflow | 方案 B：智能 Agent |
|------|---------------------|-------------------|
| **规则清晰度** | 硬编码分支，一目了然 | 黑箱决策，难以解释 |
| **开发成本** | 低，纯条件判断 | 高，需数据+模型+工程团队 |
| **合规审计** | 每笔可溯源，责任明确 | 决策不透明，纠纷难举证 |
| **边缘场景** | 死板，无法灵活处理 | 多维度融合，智能判别 |
| **人力消耗** | 中高金额+超期全人工，线性增长 | 大量自动处理，仅高风险转人工 |
| **自适应能力** | 改规则=改代码，迭代慢 | 持续学习，自适应变化 |
| **稳定性** | 极高，不受数据影响 | 存在模型漂移、误判风险 |
| **反欺诈** | 仅靠金额阈值 | 识别行为模式，精准拦截 |

### 1.4 适用场景速查

| 用 Workflow | 用 Agent |
|------------|---------|
| 规则高度标准化 | 场景复杂、边缘案例多 |
| 无算法团队 | 有数据+AI 团队 |
| 合规要求极高 | 工单量大、人力成本高 |
| 业务规模小 | 需要反欺诈精细化 |
| 政策长期不变 | 促销频繁、规则常变 |

### 1.5 方案 C：最优融合架构

**核心思想：刚性规则 Workflow 兜底 + 柔性场景 Agent 增强。**

```
用户提交退款
    │
    ├──【Workflow 层】刚性规则卡死
    │   ├── 定制品 → 拒绝（法律硬性规定，不可争议）
    │   ├── 超30天 → 拒绝（超期硬门槛）
    │   └── >5000元 → 强制转人工审批
    │
    ├──【Agent 层】柔性边缘决策
    │   ├── 低风险标准工单 → 自动通过
    │   ├── 模糊边缘工单 → Agent 综合评估
    │   └── 高风险疑似欺诈 → 标记转人工
    │
    └──【人工兜底层】
        └── Agent 决策可回溯、可纠错 → 纠错数据反哺模型
```

**方案 C 的优势：**

| 保留 Workflow 优点 | 获得 Agent 优点 | 规避两者缺点 |
|-------------------|----------------|------------|
| 合规可控 | 智能判别 | 不僵化 |
| 规则清晰 | 节省人力 | 不黑箱 |
| 可解释 | 反欺诈 | 可审计 |

## 二、TAO 循环升级：给智能体加三个新能力

### 2.1 回顾：TAO 循环基础

```
┌──────────────────────────────────────────────┐
│                                              │
│   Thought（思考）→ Action（行动）→ Observation（观察）│
│        ↑                                      │
│        └──────────── 循环 ────────────────────┘ │
└──────────────────────────────────────────────┘
```

### 2.2 功能一：长期记忆

**改什么：** 在循环外增加一个 Memory 存储器。

```
记忆 = {
    "景点类型": "历史文化",
    "预算": "中等",
    "饮食禁忌": "不吃辣"
}

循环：
    Thought → 先读取记忆，作为推理约束条件
    Action  → 按记忆偏好推荐
    Observation → 从用户对话中提取新偏好 → 更新记忆
```

**关键点：** 记忆是跨轮次持久化的，不随本轮对话结束而消失。

### 2.3 功能二：售罄自动备选

**改什么：** Action 之后增加一个结果检查节点。

```
Action  → 推荐景点 + 查询门票状态
           │
           ├── 可购 → 正常流程
           │
           └── 售罄 → Thought（触发备选逻辑）
                       └── 同类型 + 同区域 + 同预算 → 选备选
                       └── Action → 自动推荐备选
```

**关键点：** 备选必须和原推荐在关键维度上对齐（类型/预算/区域），不能乱推。

### 2.4 功能三：连续拒绝三次 → 反思策略

**改什么：** 在 Observation 中增加拒绝计数器。

```
Observation → 用户拒绝
    │
    ├── 拒绝次数 +1（记录到记忆）
    │
    └── Thought：
         ├── 拒绝 < 3次 → 微调推荐（换同类型其他景点）
         │
         └── 拒绝 == 3次 → 反思触发器
              ├── 之前策略全部作废
              ├── 换类型 / 换预算 / 换区域 / 直接询问需求
              └── 清空拒绝计数，新策略启动
```

### 2.5 升级后的完整 TAO 循环

```
┌─────────────────────────────────────────────────┐
│  Memory（长期记忆：偏好 + 拒绝计数）               │
└────────────┬────────────────────────────────────┘
             │
             ▼
    ┌── Thought ──────────────────────────┐
    │  • 读取记忆中的偏好                    │
    │  • 检查拒绝次数（≥3则换策略）            │
    │  • 选择推荐策略                        │
    └────────────┬────────────────────────┘
                 ▼
    ┌── Action ───────────────────────────┐
    │  • 推荐景点                           │
    │  • 查询门票状态                        │
    │  • 售罄 → 自动推备选                    │
    └────────────┬────────────────────────┘
                 ▼
    ┌── Observation ──────────────────────┐
    │  • 用户反馈（接受/拒绝）                 │
    │  • 提取新偏好 → 更新记忆               │
    │  • 更新拒绝计数                        │
    └────────────┬────────────────────────┘
                 │
                 └────→ 回到 Thought
```

## 三、双系统理论：用系统 1 + 系统 2 设计医疗诊断助手

### 3.1 理论基础

卡尼曼的认知双系统：

| | 系统 1（快思考） | 系统 2（慢思考） |
|--|---------------|---------------|
| 速度 | 快速、自动 | 缓慢、需努力 |
| 方式 | 直觉、模式匹配 | 逻辑推理、分析 |
| 能耗 | 低 | 高 |
| 适用 | 简单、熟悉任务 | 复杂、陌生任务 |
| 类比 | 骑自行车 | 解数学题 |

### 3.2 场景：智能医疗诊断助手

**目标：** 帮助医生初步判断病情，分流急诊 / 普通 / 慢病。

### 3.3 任务分工

**系统 1 负责 —— 快、准、无需深度推理：**

| 任务 | 为什么归系统 1 |
|------|-------------|
| 体温≥39℃ + 剧烈胸痛 → 直接标急诊 | 模式匹配，无需推理 |
| 咳嗽 + 流涕 + 低烧 → 普通感冒 | 常见症状组合，直接判定 |
| 提取年龄/性别/过敏史/既往病史 | 信息抽取，结构化任务 |
| 标准化问答（开药提醒、检查注意事项） | 模板化输出 |
| 相似病历匹配 | 向量检索，非推理 |

**系统 2 负责 —— 复杂、需逻辑链：**

| 任务 | 为什么归系统 2 |
|------|-------------|
| 疑难杂症：多种症状交叉，无明确模式 | 需要因果推理 |
| 多药联用风险评估 | 需计算药物相互作用 |
| 手术 vs 保守治疗权衡 | 多因素收益/风险分析 |
| 罕见病推理 | 需医学知识+逻辑推导 |
| 复杂病情向患者解释 | 需组织因果链条 |

### 3.4 双系统协同流程

```
患者输入
    │
    ▼
┌──────────────────────┐
│  系统 1：快速分诊      │
│  • 提取关键信息        │
│  • 识别常见模式        │
│  • 标注紧急程度        │
└──────┬───────────────┘
       │
       ├── 简单/常规病例 → 系统 1 直接输出结果
       │
       └── 复杂/疑难/高风险 ──→ 转交
                                    │
                                    ▼
                        ┌──────────────────────┐
                        │  系统 2：深度推理      │
                        │  • 多因素综合分析      │
                        │  • 鉴别诊断           │
                        │  • 治疗方案制定        │
                        │  • 生成解释           │
                        └──────┬───────────────┘
                               │
                               ▼
                    系统 1 接收结果 → 格式化输出 → 对话交互
                               │
                               └── 新知识补充到系统 1 的模式库
```

**一句话总结：系统 1 做筛分和常规，系统 2 做深度推理和风险决策，系统 2 的结论反哺系统 1，形成正向循环。**

## 四、大模型智能体的三大核心问题

### 4.1 问题一：幻觉从哪来？

**现象：** 生成看似合理但完全错误的内容。

| 根因 | 解释 |
|------|------|
| **本质是概率预测** | 模型不是"知道答案"，而是在预测"下一个词最可能是什么"。它追求的是语言通顺，不是事实正确 |
| **无事实核查能力** | 模型不知道信息真假，只能基于训练数据中的统计模式生成 |
| **训练数据有噪声** | 数据中的错误、矛盾、过时信息会被学到并复现 |
| **多轮推理累积错误** | TAO 循环中，第一步的微小偏差会在后续步骤中被放大，越推越离谱 |

**一句话：模型基于概率生成文本，没有事实验证机制，数据有噪声，多步推理会累积放大错误。**

### 4.2 问题二：不设最大循环次数会怎样？

**现象：** 陷入无限死循环。

```
Thought → Action → Observation → Thought → Action → ...
   │                                                    │
   └──────────── 永远无法收敛到终态 ──────────────────────┘
```

| 后果 | 表现 |
|------|------|
| 重复输出 | 不断生成相同或极度相似的内容 |
| 资源耗尽 | Token 消耗无限增长，成本飙升 |
| 任务卡死 | 永远无法给出最终答案 |
| 用户体验崩溃 | 无响应或输出大量垃圾 |

**原因：** 模型没有"判断何时应该终止"的内置机制。在模糊场景下，它无法说出"我不知道"或"就这样吧"，只能继续循环。

### 4.3 问题三：如何评估智能体的"智能"？

**仅用准确率？完全不够！**

| 维度 | 衡量什么问题 | 为什么重要 |
|------|------------|----------|
| **准确率** | 答案对不对 | 基础指标，但远不够 |
| **完成率** | 能不能完成，中途卡不卡死 | 卡死的智能体准确率再高也白搭 |
| **效率** | 几步完成？消耗多少 token？ | 步骤越少越聪明，成本越低 |
| **可靠性** | 幻觉率？输出稳定吗？ | 偶尔对的智能体不可用 |
| **可解释性** | 能说清楚为什么这样决策吗？ | 合规、审计、用户信任的基础 |
| **适应性** | 遇到没见过的新场景能应对吗？ | 真实世界不按训练集出牌 |
| **鲁棒性** | 输入干扰会不会崩溃？ | 用户输入不规范、有噪音是常态 |
| **安全性** | 会不会生成有害内容？ | 底线指标 |

**结论：评估智能体需要多维度的"综合体检"，准确率只是其中一项血常规。**

## 总结：一张图收尾

```
                    智能体设计的核心权衡

    Workflow ────────────────────────────── Agent
    （确定性）                                （灵活性）
       │                                        │
       ├── 刚性规则 ────────── 融合层 ── 柔性决策 ──┤
       │                                        │
       ▼                                        ▼
    合规可控                                智能高效
    成本低                                  省人力
    稳定                                    自适应
       │                                        │
       └──────────── 方案 C：各取所长 ─────────────┘

    ─────────────────────────────────────────────

    智能体能力升级三件套：
    Memory（记忆） + Fallback（兜底） + Reflection（反思）

    ─────────────────────────────────────────────

    双系统架构：
    系统 1（快）抓常规 → 系统 2（慢）啃硬骨头 → 系统 2 反哺系统 1

    ─────────────────────────────────────────────

    三大局限：
    幻觉（概率本质） + 死循环（无终止判断） + 评估不能只看准确率
```
