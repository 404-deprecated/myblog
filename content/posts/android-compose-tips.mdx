用 Compose 写了大半年，踩过不少坑。这篇主要记录几个在实际项目中遇到的性能问题，以及对应的解法。

## 问题一：不必要的重组

最常见的问题。当一个 `@Composable` 函数读取了某个 State，那它就会在 State 变化时重组。听起来很合理，但实际上很容易写出「牵一发动全身」的代码。

```kotlin
// 坏写法：整个列表项因为 isSelected 变化而重组
@Composable
fun ItemRow(item: Item, selectedId: String) {
    val isSelected = item.id == selectedId
    Row(
        modifier = Modifier.background(
            if (isSelected) Color.Blue else Color.Transparent
        )
    ) {
        Text(item.title)
        // ... 其他内容
    }
}
```

改进方案：用 `derivedStateOf` 或拆分 Composable，让颜色计算只影响需要重绘的部分。

```kotlin
// 好写法：只有背景层需要读 selectedId
@Composable
fun ItemRow(item: Item, isSelected: () -> Boolean) {
    Row(
        modifier = Modifier.background(
            if (isSelected()) Color.Blue else Color.Transparent
        )
    ) {
        ItemContent(item)
    }
}
```

注意把 `isSelected` 改成 lambda，这样 Compose 会推迟读取，减少重组范围。

## 问题二：LazyColumn 中的 key

忘记给 `items` 指定 `key`，导致列表数据变化时整个列表重组：

```kotlin
// 坏写法
LazyColumn {
    items(list) { item ->
        ItemRow(item)
    }
}

// 好写法
LazyColumn {
    items(list, key = { it.id }) { item ->
        ItemRow(item)
    }
}
```

加了 `key` 之后，Compose 能识别哪些 item 是同一个，只重组真正变化的部分。

## 问题三：remember 的粒度

```kotlin
// 每次重组都创建新的 lambda
Button(onClick = { doSomething(item.id) }) { ... }

// 用 remember 缓存
val onClick = remember(item.id) { { doSomething(item.id) } }
Button(onClick = onClick) { ... }
```

在 `LazyColumn` 这类高频重组场景下，这个细节差距明显。

## 工具推荐

- **Layout Inspector**：看重组次数，定位哪个 Composable 重组过多
- **Recomposition Highlighter**：在 debug build 里可视化看哪里在重组

这些坑踩过一次就不会忘了。Compose 整体还是比 View 系统好写很多，性能问题主要来自对重组机制理解不够深。
